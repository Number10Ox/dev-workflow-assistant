# Phase 7.2: Workflow Friction Reduction

> **Scope note (added 2026-04-29):** This phase scopes *internal-state* friction
> only (lazy parsing, stale packets, gitignore, `dwa stats`/`validate`/`clean`).
> A separate, friction-driven Phase **7.3 — Deployment-Context Decoupling** was
> created when Jon's workflow shifted to Claude Code terminal/desktop and the
> VSCode-bridge requirement blocked dogfooding. See
> `process/plan-decouple-linear-from-vscode.md`.

## Goal

Reduce manual workflow steps and maintenance burden through automation, lazy evaluation, and self-cleaning state management.

## Problem Statement

Current DWA workflow has friction points that disrupt developer flow:

1. **Manual re-parsing**: Users must remember to run `dwa parse` after spec edits
2. **Stale packets**: Packets may use outdated registry data without warning
3. **State accumulation**: `.dwa/` directory accumulates orphaned files over time
4. **Manual gitignore**: Users must manually add `.dwa/` to `.gitignore`
5. **No visibility**: No quick way to see state health or statistics

## Design Approach

### Lazy Evaluation

Instead of requiring explicit `dwa parse`, commands that need registry data check freshness:

```javascript
// Before generating packet
if (specModified > registryModified) {
  console.log('Spec has changed. Re-parsing...');
  await parseSpec();
}
```

### Staleness Warnings

Packets include freshness metadata:

```markdown
> **Registry Freshness**: Last parsed 3 hours ago. Spec modified since.
> Run `dwa parse` to update, or continue with potentially stale data.
```

### Maintenance Commands

Clean up derived state without losing important data:

```bash
dwa clean              # Remove orphaned deliverables (30+ days)
dwa clean --all        # Remove all derived state, keep source files
dwa validate           # Check state integrity
dwa stats              # Show statistics and health
```

### Gitignore Template

Auto-generate or append to `.gitignore`:

```bash
dwa init               # Creates .gitignore entry if missing
dwa --install          # Also handles .gitignore setup
```

## Technical Decisions

### File Watcher Mode (DEFERRED)

Background `dwa watch` considered but deferred to Phase 8+:
- Adds complexity (daemon management, resource usage)
- Less critical with lazy evaluation
- Can be added later without breaking changes

### State Directory Strategy

- `.dwa/` is derived state, regenerated on demand
- Should be gitignored by default
- Source of truth: `feature-spec.md`, TDDs, code files
- Registry is cache, not canonical data

### Orphan Detection

Deliverables become orphaned when:
- Removed from spec during re-parse (flagged, not deleted)
- 30+ days old with `orphaned: true` flag → eligible for cleanup

## Success Criteria

1. `dwa clean` removes old orphaned deliverables safely
2. `dwa validate` checks state integrity with actionable output
3. `dwa stats` shows deliverable counts, drift stats, disk usage
4. Packet generation warns when registry is stale vs. spec
5. `dwa --install` ensures `.dwa/` is gitignored
6. No breaking changes to existing workflows

## Out of Scope

- File watcher daemon (deferred)
- VS Code extension (separate project)
- Real-time sync (deferred)
