<feature>
  <name>Drift Event Validation (TDD)</name>
  <files>src/drift/validate-event.js, tests/drift/validate-event.test.js</files>
  <behavior>
    validateDriftEvent(event) validates required fields and enum values:

    Required fields: id, at, source, kind, summary, decision
    Optional fields: applies_to_next_work, evidence_refs, patch_proposals, author

    Valid kinds: impl_deviation, scope_change, qa_gap, spec_update_needed, tdd_update_needed, followup_required, rollback_required
    Valid decisions: pending, accept, revert, escalate
    Valid sources: complete_command, manual, skill

    Cases:
    - Valid event with all required fields -> returns true, no error
    - Missing 'id' field -> throws DWA-E070: Missing required drift field: id
    - Missing 'at' field -> throws DWA-E070: Missing required drift field: at
    - Invalid kind 'unknown' -> throws DWA-E071: Invalid drift kind: unknown
    - Invalid decision 'reject' -> throws DWA-E072: Invalid drift decision: reject
    - Invalid source 'api' -> throws DWA-E073: Invalid drift source: api
    - Extra fields allowed (forward compatibility) -> returns true
  </behavior>
  <implementation>
    Export DRIFT_KINDS, DRIFT_DECISIONS, DRIFT_SOURCES as const arrays.
    validateDriftEvent checks required fields exist, validates enums.
    Throws descriptive errors with codes DWA-E070 through DWA-E073.
  </implementation>
</feature>

<feature>
  <name>Append Drift Event (TDD)</name>
  <files>src/drift/append-event.js, tests/drift/append-event.test.js</files>
  <behavior>
    appendDriftEvent(deliverableId, eventData, projectRoot) appends to registry:

    - Reads existing registry JSON
    - Initializes drift_events array if not present
    - Generates id (crypto.randomUUID) and at (ISO timestamp) if not provided
    - Validates event via validateDriftEvent
    - Appends to drift_events (never mutates existing events)
    - Updates drift_open_count (count of pending + escalate)
    - Atomic write via writeJsonWithSchema

    Cases:
    - First drift event -> creates drift_events array, appends event
    - Second drift event -> appends after first, preserves first unchanged
    - Event with missing id -> auto-generates UUID
    - Event with invalid kind -> throws validation error, no write
    - Deliverable not found -> throws DWA-E080
    - drift_open_count updated -> pending/escalate decisions counted
  </behavior>
  <implementation>
    Use crypto.randomUUID() for event IDs (Node 14.17+ built-in).
    Read -> validate -> append -> compute derived -> atomic write.
    Never modify existing events in the array.
  </implementation>
</feature>

<feature>
  <name>Structural Drift Detection (TDD)</name>
  <files>src/drift/structural-compare.js, tests/drift/structural-compare.test.js</files>
  <behavior>
    detectStructuralDrift(registry, projectRoot) compares spec vs registry:

    Returns: { detected: boolean, kind: string|null, summary: string|null, details: object|null }

    Detection rules (in priority order):
    1. Deliverable removed from spec -> kind: spec_update_needed, summary includes "removed from spec"
    2. AC count mismatch -> kind: impl_deviation, summary includes AC counts
    3. Missing PR URL when status=completed -> kind: qa_gap, summary "no PR URL recorded"
    4. Description changed -> kind: spec_update_needed, summary "Description changed"

    Cases:
    - Spec and registry match -> { detected: false, kind: null, summary: null }
    - AC count differs (spec: 3, registry: 5) -> { detected: true, kind: 'impl_deviation', summary: 'AC count mismatch: spec has 3, registry has 5' }
    - Deliverable removed from spec -> { detected: true, kind: 'spec_update_needed', summary: 'Deliverable DEL-001 removed from spec...' }
    - Completed without PR -> { detected: true, kind: 'qa_gap', summary: 'Deliverable marked completed but no PR URL recorded' }
  </behavior>
  <implementation>
    Load feature.json to get spec_path.
    Parse spec via parseSpec (from Phase 3).
    Compare normalized fields, ignoring whitespace.
    Count AC lines (non-empty lines from acceptance_criteria field).
  </implementation>
</feature>
