    Create Handlebars template for drift log rendering with these sections:

    **Header:**
    - Title: "# Drift Log"
    - Generated timestamp
    - Open drift count and total events count

    **Section 1: Open Drift (Requires Decision)**

    - Warning indicator if applies_to_next_work = true
    - Patch proposals as bullet list if present

    **Section 2: Accepted Drift (Applied)**
    - Compact list of accepted drift: deliverable_id, timestamp, summary

    **Section 3: Reverted Drift (Rolled Back)**
    - Compact list of reverted drift: same format as accepted

    **Section 4: By Deliverable**
    - Group all events by deliverable ID
    - Each deliverable shows its event history chronologically

    Use existing Handlebars patterns from packet-v1.hbs (helpers, conditionals).
    Template receives: { generated_at, open_count, total_events, open_drift, accepted_drift, reverted_drift, by_deliverable }

  <name>Task 2: Create rebuild-log utility and command</name>
  <files>src/drift/rebuild-log.js, src/commands/rebuild-drift-log.js, tests/drift/rebuild-log.test.js, tests/commands/rebuild-drift-log.test.js</files>
  <action>
    **src/drift/rebuild-log.js:**
    - aggregateDriftEvents(deliverableDir): Reads all DEL-*.json files, extracts drift_events, adds deliverable_id to each event, returns sorted array (newest first)
    - rebuildDriftLog(projectRoot): Aggregates events, categorizes (open/accepted/reverted), groups by deliverable, renders template, writes to .dwa/drift-log.md
    - Returns: { openDrift: number, totalEvents: number, logPath: string }

    **src/commands/rebuild-drift-log.js:**
    - runRebuildDriftLog(projectRoot): Wrapper that calls rebuildDriftLog, returns structured result for VS Code extension
    - Returns: { success: boolean, openDrift: number, totalEvents: number, logPath: string, errors: [] }

    **Tests:**
    - rebuild-log.test.js: Test aggregation from multiple registries, empty registries, sorting, categorization
    - rebuild-drift-log.test.js: Test command wrapper, error handling, output structure

    Test cases:
    - No drift events -> log shows "No open drift" message
    - Multiple deliverables with drift -> correct aggregation and grouping
    - Mixed decisions (pending, accept, revert) -> correct categorization
    - Same registry rebuilt twice -> identical output (deterministic)
  </action>
