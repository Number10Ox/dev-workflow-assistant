<task type="auto">
  <name>Task 1: Create /dwa:propose-drift-patches skill</name>
  <files>skills/dwa-propose-drift-patches/SKILL.md</files>
  <action>
    Create Claude Code skill that proposes spec and TDD edits based on drift events.

    **Skill metadata:**
    - Name: dwa:propose-drift-patches
    - Description: "Propose spec/TDD updates based on implementation drift"
    - disable-model-invocation: false (LLM-powered skill)

    **Inputs (via user prompt or context):**
    - Deliverable ID (required)
    - Optional: PR URL, commit SHA, or git diff

    **Process (5 steps):**
    1. Load deliverable registry (.dwa/deliverables/DEL-###.json)
    2. Extract drift_events (filter to pending/escalate decisions)
    3. Load linked spec (feature-spec.md) and TDD (if exists)
    4. Analyze drift summary + evidence + current contract text
    5. Generate patch proposals for:
       - Spec deliverable row (AC additions/changes, description updates)
       - TDD guardrails/decisions (if implementation revealed new constraints)
       - QA notes (if test strategy changed)

    **Output format:**
    ```markdown
    ## Proposed Patches for DEL-###

    ### Drift Events Analyzed
    - [D1] impl_deviation: Added error handling not in ACs

    ### Spec Updates (feature-spec.md)

    **Acceptance Criteria additions:**
    - [ ] System handles connection timeout with retry (3 attempts)
    - [ ] Error message displayed to user on final failure

    **Description update:**
    > Add: "Includes robust error handling for network failures"

    ### TDD Updates (docs/tdds/feature-tdd.md)

    **Guardrails addition:**
    - Network calls must use timeout wrapper with configurable duration

    **Decision log addition:**
    - [2026-01-24] Added retry logic for transient failures (3 attempts with exponential backoff)

    ---
    *Review these proposals and apply manually to spec/TDD files.*
    ```

    **Key behaviors:**
    - Never auto-apply changes (proposals only)
    - Reference specific drift event IDs in proposals
    - If no drift events found, report "No pending drift for this deliverable"
    - If PR URL provided, suggest using git diff for detailed analysis
  </action>

<task type="auto">
  <name>Task 2: Create /dwa:summarize-drift skill</name>
  <files>skills/dwa-summarize-drift/SKILL.md</files>
  <action>
    Create Claude Code skill that produces human-readable drift summaries.

    **Skill metadata:**
    - Name: dwa:summarize-drift
    - Description: "Generate human-readable drift summary for PR or stakeholder"
    - disable-model-invocation: false (LLM-powered skill)

    **Inputs (via user prompt):**
    - Scope: "all" (project-wide) or specific deliverable ID
    - Audience: "technical" (for PR) or "stakeholder" (for PM/product)

    **Process (4 steps):**
    1. Load drift log (.dwa/drift-log.md) or specific deliverable registry
    2. Filter to relevant drift (open, or recent if scope=all)
    3. Adapt language for audience (technical vs stakeholder)
    4. Generate summary with actionable next steps

    **Output format (technical, for PR):**
    ```markdown
    ## Drift Summary

    **Scope:** DEL-001 (User Authentication)
    **Open items:** 2

    ### Implementation Deviations

    1. **Added retry logic** - Network calls now include 3-attempt retry with exponential backoff. Not in original ACs but improves reliability.
       - Decision: Pending
       - Recommendation: Accept and update spec

    2. **Changed error format** - Errors now return structured {code, message} instead of string.
       - Decision: Pending
       - Recommendation: Accept (breaking change documented)

    ### Next Steps
    - [ ] Review patches: Run `/dwa:propose-drift-patches DEL-001`
    - [ ] Update spec with accepted changes
    - [ ] Close drift events with decisions
    ```

    **Output format (stakeholder, for PM):**
    ```markdown
    ## Implementation Update: User Authentication

    During implementation, we made two changes from the original plan:

    1. **Improved reliability** - Added automatic retry for network issues. Users will see fewer connection errors.

    2. **Better error messages** - Error responses now include specific codes for easier debugging.

    **Impact:** Both changes improve the user experience. No timeline impact.

    **Decision needed:** Please confirm these changes are acceptable, or flag if we should revert.
    ```

    **Key behaviors:**
    - Adapt tone and detail level for audience
    - Highlight decisions needed vs informational updates
    - Include actionable next steps
    - Reference specific deliverable IDs
  </action>
