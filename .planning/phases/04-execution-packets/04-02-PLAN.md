  <action>
Create `skills/dwa-enrich-packet/SKILL.md` following the pattern from existing skills (dwa-create-spec, dwa-draft-tdd).

The skill should:

1. **Header frontmatter:**
```yaml
---
name: enrich-packet
description: Enrich an execution packet with implementation targets, file suggestions, and risk notes from codebase analysis. Does NOT modify contract sections (ACs, QA, constraints).
disable-model-invocation: false
argument-hint: [deliverable-id]
---
```

Note: `disable-model-invocation: false` because this skill USES the LLM for codebase analysis and content generation.

2. **Process steps:**

**Step 1: Locate packet**
- Read packet from `.dwa/packets/{deliverable-id}.md`
- If packet doesn't exist, error: "Packet not found. Run /dwa:start first."
- Extract key context: goal, user story, acceptance criteria, constraints

**Step 2: Analyze codebase**
Use Claude's tools to search the codebase:
- Grep for keywords from goal and AC descriptions
- Glob for file patterns related to the feature area
- Read relevant files to understand existing patterns

**Step 3: Generate enrichment sections**
Produce content for:

a) **Implementation Targets** (Section 6 addition):
   - Files likely touched (existing files with grep matches)
   - New files expected (based on naming conventions)
   - Key APIs to use (from existing code patterns)
   - Key APIs to avoid (anti-patterns, deprecated)

b) **Gotchas & Risk Notes:**
   - Lifecycle constraints (if applicable)
   - Threading considerations
   - Serialization edge cases
   - Platform-specific issues

c) **Test Additions:**
   - Where unit tests live (detected from codebase)
   - Test harness conventions
   - Recommended test coverage areas

d) **AC Coverage Map:**
   - For each AC, suggest: unit test / integration test / manual check
   - Flag ACs that may be hard to test automatically

**Step 4: Append enrichment to packet**
Add new sections AFTER the existing content:

```markdown
---

## ENRICHMENT (LLM-Generated)

> The following sections were added by /dwa:enrich-packet and contain AI-assisted analysis.
> Contract sections above (Goal, ACs, QA, Constraints) are authoritative and unchanged.

### Implementation Targets

**Files Likely Touched:**
- [files from analysis]

**New Files Expected:**
- [files based on conventions]

**Key APIs:**
- USE: [recommended patterns]
- AVOID: [anti-patterns]

### Gotchas & Risk Notes

[context-specific warnings]

### Test Additions

**Unit Tests:** [location]
**Integration Tests:** [location]
**AC Coverage Map:**
| AC | Test Type | Notes |
|----|-----------|-------|
| C1 | Unit | [suggestion] |

### Patch Proposal

> If contract changes are recommended, list them here. User decides whether to accept.

[Any suggested changes to ACs, QA, or constraints go HERE, not inline]
```

**Step 5: Confirm enrichment**
Report to user:
- "Packet enriched: .dwa/packets/{id}.md"
- "Added sections: Implementation Targets, Gotchas, Test Additions, AC Coverage Map"
- "Contract sections (Goal, ACs, QA, Constraints) unchanged"
- ""
- "Review the Patch Proposal section if any contract changes are suggested."

3. **Critical rules for the skill:**
- MUST NOT edit text in sections 0-8 (control, guardrails, goal, story, ACs, QA, dependencies, provenance, drift)
- MUST NOT remove or reorder existing content
- Contract changes ONLY in Patch Proposal section
- All enrichment goes in clearly marked "ENRICHMENT" section
- Enrichment is additive, never destructive
